## Verdict

**APPROVED WITH REQUIRED FOLLOW-UP**

B4 and B5 are closed for every row shape and probe the specification enumerates, with correct fail-closed semantics, no observed over-restriction, and a demonstrably effective fix (both leaks reproduced on the baseline, absent after the commit). One residual leak in the same corrupt-journal threat class was found outside the spec's enumerated shapes (F1 below): a corrupt `artifact_links` row whose URI names a record with a case- or format-variant scheme (`Record://<id>`, `record:/<id>`) bypasses target extraction and returns the foreign record id verbatim inside `artifact_uri` through both `get_related` and `search(include_artifacts=True)`. It does not grant access to the record itself and requires the same direct DB corruption B4 required, but under the spec's own defense-in-depth rationale it must be closed before the read side can be called uniformly fail-closed. Package 6 does not depend on this path, so it does not block sequencing.

## Threat model

Same as B4's: the read side must not assume a clean journal. The adversary (or entropy) is a corrupt/legacy journal row — a pointer the current write path would refuse to create — plus any future library caller who omits scope. The caller is a scoped integration identity (workspace grants, optional sensitive grants) reading through `Brain.get_record` / `get_related` / `search`, directly or via MCP. Assets: foreign-workspace record ids and any distinguishing detail about out-of-scope, sensitive-without-grant, dangling, or non-visible-status targets. The journal schema itself (NOT NULL constraints, table shapes) is assumed intact; an adversary who can rewrite the schema is out of scope.

## B4 status — CLOSED (with F1 residual in the same class)

- `Repository.related()` produces exactly five row shapes; the only DB-controlled fields that drive extraction are `relation`, `artifact_uri`, and `record_id` (`direction` is set by repository code, never read from the DB). `Brain._relation_target_id` handles: supersede rows by relation name, incoming rows by direction, outgoing `record://` rows by URI prefix. For all three, the extracted id is the actual other record, and every extracted id flows through the single `Brain._target_visible` check — target exists, status not candidate/rejected/forgotten, workspace in scope, sensitivity covered — identical rules to a direct `get_record`.
- Provenance sanitization covers both declared pointer surfaces: `RecordEnvelope.supersedes/superseded_by` (`get_record`) and `Provenance.supersedes/superseded_by` (`search`), nulled via the same `_target_visible` check. A search of all response models (`brain/models.py`) found no other record-pointer field: `journal_record_id` and `source_event_id` are the record's own identifiers; `SearchResult.artifact_links` rows go through `_scope_related`; `RelatedResponse.related` rows likewise; audit lines log only the caller's own requested/returned primary ids. No leak through row counts, ordering, placeholders, or error details (out-of-scope, dangling, and hidden-status targets are all indistinguishable `BRAIN_RECORD_NOT_FOUND` on direct fetch; sensitive-without-grant is `BRAIN_SENSITIVE_SCOPE_DENIED`, matching the pre-existing direct `get_record` contract exactly).
- Canonical journal state is never modified — verified by SQL read-back after sanitization.
- **Residual:** F1 (below) — record ids smuggled inside non-canonical `record:`-scheme spellings of `artifact_uri` bypass extraction and are returned verbatim. This is the one remaining way a corrupt row can name a foreign record without being resolved.

## B5 status — CLOSED

- `allowed_workspaces` is a mandatory keyword-only parameter on `get_record` and `get_related`; omission raises `TypeError` deterministically before any repository or journal access. `search` is equivalently fail-closed one layer down: `_request` raises `BRAIN_UNKNOWN_WORKSPACE` for absent/empty/`"*"` workspaces before the embedding call (asserted with a counting transport).
- `None` is no longer a meaningful scope value anywhere: passing `allowed_workspaces=None` fails closed with `TypeError` (`set(None)`), and an empty list denies everything (`workspace in set([])` is always false). `sensitive_workspaces`'s default changed from `None`-meaning-unrestricted to `()` — the narrower fail-open path inside the sensitivity check is also closed; `sensitive_allowed=True` alone grants nothing without an explicit `sensitive_workspaces` entry (probed).
- `UNRESTRICTED_OPERATOR_SCOPE` is explicit, module-level, grep-visible, non-`None`, and compared by **identity** — a copied/reconstructed instance fails closed (probe E). It is unreachable from MCP: all three read tools pass `policy.resolve_scope()` output (always a sorted list of strings), MCP inputs are JSON and cannot encode the sentinel object, and `brain/mcp_server.py` contains no dynamic import, alias, wrapper, or helper that could forward it (full-file read; the only scope sources are `resolve_scope`/`authorize`). The guard test asserts both source-text absence and module-namespace absence.
- **TypeError vs BrainError for omitted scope:** acceptable for this internal API. It is deterministic, occurs before any data access, and is a programming error, not a runtime condition — a stable `BrainError` would not materially improve safety. Two caveats worth recording: (a) MCP's `_error` handler re-raises non-`BrainError`/`ValidationError` exceptions, so if a future MCP change ever did omit scope, the `TypeError` would surface as an unhandled server error rather than a typed error payload — currently unreachable, acceptable; (b) the changelog/docs correctly describe the semantics. No change required.

## I10 assessment

The property test (`test_i10_every_expanded_id_is_directly_fetchable_with_same_scope`) builds one fixture mixing a legitimate incoming link with a cross-workspace supersedes, a foreign incoming link, a dangling outgoing link, and a planted rejected record; it collects ids from `get_related` rows **and** every field of a `search(include_artifacts=True)` response (result ids, both provenance pointers, `artifact_links` record ids), asserts the corrupt set is disjoint, and replays a direct identically-scoped `get_record` for every returned id. Re-run: passes. Baseline behavior (probe 1b) confirms the property was violated pre-fix.

Completeness critique:

- It inspects nested relation rows and provenance pointers, not just top-level ids — good.
- It does **not** scan `artifact_uri` strings for record-id substrings, which is exactly where F1 leaks; the raw-substring assertion in `test_b4_probe_include_artifacts_and_provenance_sanitized_in_search` would catch F1's shape only if the fixture planted a case-variant URI, which it does not. The follow-up fix should extend the I10 collector (or the substring check) to the serialized response as a whole with all corrupt ids, including scheme-variant plants.
- The direct-fetch replay does not recurse into the envelopes it fetches (their own supersedes pointers). Low risk — those fields pass through the same `_target_visible` — but a fixpoint iteration would make the property airtight. Optional.
- The MCP-sentinel guard is source-text + `hasattr`. It fails loudly (FileNotFoundError) if `mcp_server.py` is renamed — fail-closed against that refactor — but it would not cover tool bodies moved into a helper module. See F3.

## Findings by severity

**F1 — MEDIUM (required follow-up): scheme-variant `record:` URIs in corrupt `artifact_links` rows leak foreign record ids verbatim.**
`Repository.related()` (`brain/repository.py:56`) and `Brain._relation_target_id` (`brain/api.py:136`) both key on the case-sensitive exact prefix `record://`. A corrupt row with `artifact_uri = "Record://<foreign-id>"` or `"record:/<foreign-id>"` (URI schemes are case-insensitive per RFC 3986) is not recognized as naming a record: no target is extracted, `_target_visible` never runs, and the row — foreign id embedded in the URI — is returned to a scoped caller by `get_related` and `search(include_artifacts=True)` (probes A, B, D). The write path cannot create such rows (`URI_RE` admits only six lowercase non-record schemes), so the trigger is direct DB corruption — the same threat model under which B4 was classified a blocker. The leak is the id only (direct fetch still fails), which is why this is medium, not high. **Required fix (Package 5 follow-up, before or alongside Package 6):** treat any `artifact_uri` whose scheme does not case-insensitively match the allowed write-side vocabulary as fail-closed (drop the row), or at minimum detect `record:`-scheme spellings case-insensitively/format-tolerantly and route them through `_target_visible`; extend the I10/substring tests with scheme-variant plants. Probe F (relation name `supersedes` on an outgoing artifact row selecting the source id, making the visibility check vacuous) should be handled by the same malformed-row rule.

**F2 — LOW (documentation): changelog misstates sentinel usage.**
The Package 5 changelog entry says `UNRESTRICTED_OPERATOR_SCOPE` is "used in `scripts/smoke_brain_consumer.py` only where the target record's workspace cannot be known in advance". In the committed code the smoke script uses explicit lists (`["ai-pos"]`, `["personal"]`) on both calls and the sentinel appears nowhere outside `brain/api.py` and the tests (repo-wide grep). The actual state is *stricter* than documented — but the spec changelog is the authority record and should be corrected.

**F3 — LOW (test robustness, recommended follow-up): the sentinel guard would not survive a tool-extraction refactor.**
`test_mcp_server_source_never_references_unrestricted_operator_scope` reads only `brain/mcp_server.py` and checks `hasattr` on that one module. It fails loudly if the file is renamed (good), and the `hasattr` arm is nearly redundant with the source-text arm. But if tool implementations were refactored into a helper module (e.g. `brain/mcp_tools.py`), neither arm would scan it. Recommended: an AST/import-walk assertion over every module transitively imported by `brain/mcp_server.py` (excluding `brain.api`, where the sentinel is defined, and whitelisted operator scripts), asserting no `Name`/`Attribute` reference to `UNRESTRICTED_OPERATOR_SCOPE` and no `from brain.api import *`. Not blocking: today's file contains no indirection at all, and an evasion via string concatenation is deliberate sabotage, outside what a guard test defends against.

**F4 — INFO: sensitivity-grant surface differs between `search` and `get_record`/`get_related`.**
`search` derives `sensitive_workspaces` as "all requested workspaces if `sensitive_allowed`"; the direct methods take an explicit `sensitive_workspaces` list. At the MCP layer this is consistent (`authorize` requires requested ⊆ sensitive grants before `sensitive_allowed` search runs; `get_record`/`get_related` pass the grants set). At the library layer, "identical scope" across the two APIs requires the caller to map `sensitive_allowed=True` → `sensitive_workspaces=workspaces`, which the I10 test does. No leak in either direction; recorded so a future caller doesn't misread the asymmetry as a gap.

**F5 — INFO (performance, not safety): `_target_visible` opens a fresh read-only SQLite connection per relation row** (`Repository.journal_row` → `Repository.journal()` context per call), so a `get_related` on a record with N relations performs N+1 connection cycles, and `search(include_artifacts=True)` multiplies that by result count. Correctness is unaffected; worth batching if related-row volume grows.

## Required follow-up verification (2026-07-16)

**Root cause.** `Repository.related()` (`brain/repository.py:56`, pre-fix) and `Brain._relation_target_id` (`brain/api.py:136`, pre-fix) both keyed on the case-sensitive exact prefix `"record://"` via `str.startswith`. Any `artifact_links.artifact_uri` value that named a record but didn't match that literal prefix byte-for-byte fell through both checks: extraction returned `None`, `_target_visible` never ran, and the row — including its raw `artifact_uri` text with the foreign id embedded — was passed straight through `_scope_related` to the caller, indistinguishable from a genuine non-record artifact link.

**Fix — centralized fail-closed classification.** New module `brain/record_uri.py` exports `classify_record_uri(uri)`, the single source of truth used by both `Repository.related()` and `Brain` (`_classify_relation_row`, replacing `_relation_target_id`). It returns exactly one of three values, in this order:

1. **`canonical_record_target`** — the raw string fullmatches `^record://[^/\s?#]+$`: lowercase scheme, exactly two slashes, a non-empty id with no further `/`, whitespace, `?`, or `#`. Only this shape is resolved and routed through the existing `_target_visible` scope check — unchanged from the pre-follow-up behavior.
2. **`malformed_record_like`** — not canonical, but percent-decoding + stripping + lowercasing the string yields something starting with `"record:"`. This is the fail-closed bucket: `_scope_related` drops the row outright (`continue`), before it ever reaches a response model. It is never resolved (no tolerant case-insensitive matching to a real target) and never falls through to the normal-artifact path (so its raw URI text, with the embedded foreign id, is never serialized).
3. **`normal_artifact`** — everything else, including any URI that merely contains the substring "record" without a `record:`-scheme prefix (e.g. `doc://records-overview`) — behavior for this class is byte-for-byte unchanged.

The classifier is intentionally narrow: detection is anchored at the start of the (decoded, stripped) string, not a substring search, so it cannot misclassify unrelated artifact URIs. `Brain._classify_relation_row` also closes the review's probe F: a genuine `record_state.supersedes`/`superseded_by` row (built by `Repository.related()` without an `artifact_uri` key at all) is still matched by relation name; a corrupt `artifact_links` row that merely *reuses* the relation name `"supersedes"` now falls through to URI classification like any other outgoing row, instead of vacuously passing the visibility check on the row's own self id.

**Variant matrix — all pass (`tests/test_brain_scope_integrity.py::test_f1_malformed_record_like_uri_dropped_from_get_related_and_search`, 10 parametrized cases).** For each of `Record://<id>`, `RECORD://<id>`, `record:/<id>`, `record:////<id>`, `" record://<id>"`, `"record://<id> "`, `record://` (empty id), `record://<id>?x=1`, `record://<id>#fragment`, and a percent-encoded spelling (`record%3A%2F%2F<id>`): the foreign id and the raw malformed URI are both absent from `get_related()` and from `search(include_artifacts=True)`; the malformed row is dropped entirely (only the fixture's pre-existing legitimate `evidence` link remains — asserted by exact list equality, not just id absence); the journal file is byte-identical before/after (read-only enforced by `PRAGMA query_only=ON`); and no `BrainError.details` echoes the foreign id. Positive regressions confirm canonical `record://<same-workspace-id>` still resolves and returns when in scope (`test_canonical_record_relation_resolves_within_scope`), the same canonical shape pointed cross-workspace is still filtered by the pre-existing B4 logic (`test_canonical_record_relation_to_foreign_workspace_still_filtered`), and all six non-record schemes (`repo`, `git`, `doc`, `adr`, `route`, `workspace`) pass through unaffected (`test_non_record_schemes_remain_normal_artifacts_unaffected`, parametrized).

**I10 property test extension.** `collect_ids_from_related`/`collect_ids_from_search` now also call a new `collect_uri_derived_ids()` that independently re-derives target ids by scanning `artifact_uri` text through `classify_record_uri` — not just reading the pre-populated `record_id` field — so a regression in `Repository.related()`'s own extraction would still be caught. `test_i10_every_expanded_id_is_directly_fetchable_with_same_scope` now also plants two scheme-variant corrupt rows (`Record://<foreign-id>`, `record:/rec-does-not-exist`) alongside the original canonical corrupt shapes, and adds a raw-substring scan over the full serialized `related` + `search` response text for every forbidden id — closing the exact gap the original review's I10 assessment flagged (structured-field collection alone would never see a leak that never populates a structured field).

**Sentinel guard (F3, recommended).** Added `test_mcp_transitive_imports_never_reference_unrestricted_operator_scope`: an AST-based walk of every `brain.*` module reachable by import from `brain/mcp_server.py` (transitively, excluding `brain.api` where the sentinel is legitimately defined), asserting no `Name`/`Attribute` node named `UNRESTRICTED_OPERATOR_SCOPE`, no `from brain.api import *`, and no aliased import (`from brain.api import UNRESTRICTED_OPERATOR_SCOPE as X`) anywhere in that graph. Today the graph is `{config, control, errors, instance_identity, models, write_policy, mcp_server}` — no indirection, consistent with the original review's finding. The walker (~50 lines) stayed small enough to implement directly rather than deferring as a Low follow-up; the pre-existing text-only test is kept alongside it as a cheap smoke check. Known residual limit, same as the review's own caveat: a binding rebind two levels deep (e.g. re-exporting a *different* name that happens to alias the sentinel value at runtime without any literal `UNRESTRICTED_OPERATOR_SCOPE` token appearing anywhere in source) is undetectable by any static/source-level guard and is out of scope, per the review, as deliberate sabotage.
